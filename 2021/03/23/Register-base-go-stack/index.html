Register Base go Stack

这是什么

这个设计的目的是将原本基于Stack的函数调用协定改写为基于Register寄存器的调用协定，自一开始go发布到现在，就一直使用Plan 9 ABI的基于堆栈的调用约定，其中参数和结果值通过堆栈上的内存传递。
好处是，调用约定的规则很简单，并且建立在现有结构布局规则的基础上；所有平台都可以使用基本上相同的约定，所以比较容易编写跨平台，可移植的编译器和运行时代码，这也简化了go和defer语句以及反射的实现。此外，当前的go ABI还没有使用寄存器传递函数参数，也就意味着在函数调用中没有寄存器内容存在，而使用堆栈调用协定，在调用之前，函数中的任何活动状态都必须刷新到堆栈中。所以stack的方式简化了在GC，堆栈增长以及展开的tracing&&profiling。

为什么要这么改

go的基于堆栈的调用约定在桌面上留下了很多性能。尽管现代高性能CPU在很大程度上优化了堆栈访问，但是访问寄存器中的参数仍比访问堆栈中的参数快40％。此外，基于堆栈的调用约定，尤其是没有被调用者保存寄存器的调用约定，会引起额外的内存通信，这会对总体性能产生次要影响。

有什么痛点

golang在官方issue 中，给出了目前在主线分支中针对stack base做出的修改。目前只是在amd64平台上进行最小修改，后面继续再扩展到其他平台上。

但是这里有两个比较重要的问题需要解决，第一个是goroutine本身的缩扩容是因为go runtime可以动态调整堆栈大小，这里不仅给跨平台ABI提出了棘手的设计，也给ebpf追踪go调用提高了难度，而第二个问题是goroutine本身是由go runtime调度，而不是OS内核进行调度，这意味着非go代码和go代码中的转换如果是通过平台ABI可以无缝切换。

现有的大多数ABI都是基于C语言的，它在重要方面与go有所不同。例如，大多数ELF ABI（至少x64-64，ARM64和RISC-V）会强制go片在堆栈上而不是在寄存器中传递，因为片头是3个字。同样，由于C函数很少返回一个以上的字，因此大多数平台ABI最多为结果保留两个寄存器。由于go函数通常至少返回三个字（结果和两个字的错误接口值），因此平台ABI将强制此类函数在堆栈上返回值。影响平台ABI的其他因素包括C中的数组参数通过引用而不是通过值传递，并且C中的小整数类型被提升为int而不是保留其类型。因此，平台ABI根本不适合go语言。

目前是如何设计

  • Proposal中建议每种体系结构都应定义一个整数和浮点寄存器（以及将来可能的矢量寄存器）序列，再加上大小和对齐约束。
  • 支持多字返回值。
  • 支持在堆栈增长时扫描和调整寄存器参数中的指针。
  • 一流的通用调用框架表示形式。 go和defer语句以及反射调用需要将调用帧作为的内存对象进行操作的第一公民。
  • No callee-save registers。
  • be compatible with platform frame-pointer unwinding rules。

值得注意的是，go版本之间没有二进制兼容性。这对于C的共享库很重要，但是go要求一个进程中的所有共享库都使用相同的go工具链版本。

详细设计

go, defer and reflection calls

  1. go和defer调用的类型是静态已知的，而反射调用则不是。这意味着编译器可以静态生成用于解go和defer调用参数的桥，但这不是反射调用的玩法。
  2. go和defer调用的返回值始终被忽略，而反射调用必须捕获结果。这意味着进行或延迟呼叫的呼叫桥可以是尾部呼叫，而反射呼叫可能需要按组返回值。
  3. go和defer呼叫帧是长期存在的，而反射呼叫帧是瞬态的。这意味着垃圾收集器必须能够扫描go和推迟调用帧，而我们可以将不可抢占的区域用于反射调用。
  4. go调用框架直接存储在堆栈中，而defer和reflect调用框架可以在堆中构造。这意味着垃圾回收器必须能够为go调用帧构造适当的堆栈映射，但是defer和reflect调用帧可以使用堆位图。这也意味着需要堆栈参数的延迟和反射调用必须将调用帧的那一部分从堆复制到堆栈。

设计的调用帧如下所示

struct {
    pc           uintptr          // PC of target function
    nInt, nFloat uintptr          // # of int and float registers
    ints         [nInt]uintptr    // Int registers
    floats       [nFloat]uint64   // Float registers
    ctxt         uintptr          // Context register
    stack        [...]uintptr     // Stack arguments/result space
}

go调用可以在新的goroutine堆栈上构建此结构，而调用桥可以从堆栈中弹出此结构的寄存器部分，仅将堆栈部分保留在堆栈上，并进行尾调用pc。垃圾收集器可以通过检查调用帧中的pc来识别此调用桥并构建堆栈映射。

defer和reflect调用可以使用适当的堆位图在堆中构建帧。在这种情况下，调用桥必须打开一个新的堆栈帧，将堆栈复制到堆栈中，加载寄存器参数，调用pc，然后将寄存器结果和堆栈结果复制回堆中帧（必要时使用写屏障） ）。在仅寄存器反射调用的情况下，网桥可以将寄存器参数作为参数本身，并返回寄存器结果作为结果。这可以避免任何复制或写入障碍。
